# 设置与运维

设置页不是一个简单的“个性化偏好”页面，而是 WeRSS 的运行控制台。它混合了前端偏好、后端配置覆盖、调度器状态和运维相关开关。

## 1. 这块功能到底管什么

当前“设置与运维”主要涵盖 4 类内容：

- 前端显示偏好
- 采集和标签相关配置
- 文章存储与清理策略
- 调度器和消息任务的运行状态

这也是为什么当前设置页会显得比普通后台更重：它不是单纯 UI 设置，而是部分运行控制面板。

## 3. 当前设置页包含哪些区域

### 应用偏好

这部分更偏前端本地设置，例如：

- 主题模式
- 水印开关

这类配置通常不会写回系统核心配置，只影响当前浏览体验。

### 采集起始时间

这是一个非常关键的业务配置。它会影响系统估算抓取页数和历史回补范围。

在任务侧，`jobs/mps.py` 会读取 `collect_start_date`，据此估算要抓多少页。

所以这个字段不是展示配置，而是真正影响采集行为。

### 标签提取 Prompt

设置页里现在维护了多套 Prompt 预设：

- 科技
- 医学
- 法律
- 自定义

对当前项目来说，这一块非常重要，因为标签提取质量直接影响：

- 标签列表质量
- 标签聚类效果
- 热点发现辅助信号

### 文章图片转存

这部分控制 MinIO 可用时是否转存公众号文章图片。

它不是纯前端行为，而是对象存储策略开关。

### 文章保留策略

设置页支持控制：

- 是否启用定期清理
- 保留天数
- 以 `created_at` 还是 `publish_time` 作为基准

这部分最终会影响后台清理任务注册和实际删除逻辑。

### 调度器与消息任务状态

设置页会读取系统信息接口，展示：

- 采集调度器状态
- 补全文调度器状态
- 已配置消息任务

这让它具备了运维视角，而不只是配置表单。

## 4. 配置优先级必须怎么理解

这块最容易被写错，也最容易让使用者误解。

不要简单说“页面保存后就生效”。真实情况是：

- `config.example.yaml` 只是示例
- `config.yaml` 是文件配置
- `.env` 是环境变量源
- 数据库 `config_management` 是控制台覆盖源
- 运行时环境变量可能再覆盖一次

具体优先级请看 [配置与优先级](../dev/configuration-and-precedence.md)。

在文档里必须强调：

- 设置页写入的不是唯一配置源
- 某些配置可能被环境变量覆盖

## 5. 为什么它属于“运维”

因为这页已经不只是给普通用户调偏好的页面了。它还承担了这些运维判断：

- 调度器有没有运行
- 消息任务是否已经生效
- 文章保留策略是否会自动清理数据
- MinIO 图片策略是否开启
- 当前 Prompt 配置是否正确

如果线上出现“任务不跑”“清理不生效”“图片没转存”，设置页和系统信息页是第一批要看的地方。

## 6. 常见误区

### 误区一：保存配置后所有行为都会立刻变

不对。取决于配置类型：

- 前端本地设置通常立即生效
- 后端配置可能需要重新读取
- 调度器相关配置有时需要重新注册任务

### 误区二：设置页等于配置文件编辑器

不对。它是“控制台覆盖层”，不是完整替代 `config.yaml`。

### 误区三：这里都是低风险改动

不对。文章清理、抓取起始时间、Prompt、对象存储开关都可能直接影响生产行为。
