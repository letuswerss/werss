# 扩展指南

本文档给要继续扩展 WeRSS 的开发者用。重点不是“能不能加功能”，而是“按当前项目边界，应该从哪里下手才不会把系统越改越乱”。

## 1. 扩展前先分类型

先判断你要加的是哪一类能力：

- 新 API
- 新页面
- 新后台任务
- 新分析模块
- 新配置项
- 新导入导出格式

不同类型落点不同，不要默认一上来就在 `apis/` 里开新文件。

## 2. 新增一个 API

### 什么时候应该加 API

只有当能力需要被前端或外部程序主动调用时，才值得加 API。

例如：

- 新增一个“手动重建”入口
- 新增一个列表查询接口
- 新增一个导出接口

### 基本步骤

1. 确认归属领域
2. 在 `apis/` 下补或新建文件
3. 业务逻辑尽量写到 `core/`
4. 在 `web.py` 注册 router
5. 前端在 `web_ui/src/api/` 补调用封装

### 例子

- 文章相关：`apis/article.py`
- 标签相关：`apis/tags.py`
- 消息任务：`apis/message_task.py`
- 热点发现：`apis/hot_topics.py`

如果只是现有领域里的一个补充动作，优先放进原文件，不要轻易拆新模块。

## 3. 新增一个页面

页面扩展建议按下面顺序：

1. 明确页面属于哪个导航组
2. 新建 `views/*.tsx`
3. 在 `router/index.tsx` 注册路由
4. 在 `api/` 层补接口封装
5. 需要时把入口接到 `BasicLayout`

新增页面时最容易犯的错是：

- 先写页面，后补接口
- 结果页面结构是按想象写的，和后端返回对不上

更稳的做法是：

- 先确认 API 数据形状
- 再写页面

## 4. 新增一个分析能力

像标签聚类、热点发现这类能力，不应该直接写成“某个接口里的一段大函数”。

推荐边界：

- `core/`
  放分析逻辑本体
- `jobs/`
  放定时运行
- `apis/`
  放手动触发和结果查询
- `web_ui/`
  放展示页

这样同一个能力才能支持：

- 手动跑
- 自动跑
- 结果回溯

## 5. 新增一个定时任务

建议流程：

1. 先把业务逻辑写到 `core/`
2. 在 `jobs/` 注册 cron 或后台任务
3. 在系统信息 / 设置页补状态展示
4. 如果用户需要手动触发，再补 API

不要把完整业务都堆到 `jobs/*.py` 里。`jobs/` 更适合“触发器”，不是“业务主仓库”。

## 6. 新增一个配置项

改配置必须同步考虑 4 个地方：

1. `config.example.yaml`
2. `.env.example`
3. `core/config.py`
4. 如果需要控制台编辑，再加到 `apis/config_management.py` 或设置页

还要判断这个配置是：

- 启动时读取一次
- 还是运行时可热更新

如果涉及调度器，通常保存配置后还要刷新调度器状态。

## 7. 新增一个导入导出能力

这类能力当前主要在 `apis/export.py`。

建议方式：

- 先定义导出的数据形状
- 确认权限与下载方式
- 再决定是 CSV、OPML、JSON 还是别的格式

新增导出时，不要只考虑“导出”，还要考虑：

- 是否有对应导入
- 字段能否回灌
- 临时文件是否要自动清理

## 8. 什么时候该复用，什么时候该新建

### 优先复用

- 已有领域 API 文件
- 已有数据模型
- 已有页面布局
- 已有任务调度器
- 已有 `success_response` / `error_response`

### 可以新建

- 明显是新领域
- 旧文件已经过大且边界清楚
- 新能力有稳定独立的数据模型和页面

## 9. 一个完整功能怎么落地

以“新增一个内容分析页”为例，完整落地通常是：

1. `core/` 写分析逻辑
2. `core/models/` 补结果实体
3. `apis/` 提供结果列表和重建接口
4. `jobs/` 注册定时任务
5. `web_ui/src/api/` 补前端接口
6. `web_ui/src/views/` 补页面
7. `router/index.tsx` 注册路由
8. `docs/` 补功能文档和 UI 文档

如果缺其中任意一层，后面通常都会返工。

## 10. 最容易把系统搞乱的几种方式

- 直接在 API 里写完整业务。
- 页面里直接写请求逻辑和数据清洗。
- 任务逻辑与手动逻辑各写一份。
- 新增配置但不写默认值和优先级说明。
- 新增字段但不补文档和导出层。

## 11. 建议的扩展顺序

如果你在做一个新功能，建议按下面顺序推进：

1. 先确认数据模型
2. 再写业务逻辑
3. 再暴露 API
4. 再接前端
5. 最后补文档和运维说明

这个顺序虽然不花哨，但返工最少。
