# 订阅管理

订阅管理是 WeRSS 的数据源入口。系统后面的大部分能力都建立在“先有公众号订阅，再有文章采集”这条链路上。

## 1. 这个功能真正管理的是什么

虽然页面叫“订阅管理”，但当前代码里它的核心实体实际上是 `Feed`，也就是公众号在系统中的订阅表示。

它不是泛化意义上的 RSS 订阅器，而是：

- 保存公众号基础信息
- 保存采集关联标识
- 作为文章抓取的入口
- 作为 RSS 输出的上游实体

## 3. 订阅管理承担的能力

### 公众号搜索

用户可以根据公众号名称或关键字搜索公众号。搜索不是查本地库，而是通过微信侧能力去查候选公众号。

### 添加订阅

当前支持的思路包括：

- 通过文章链接提取公众号信息
- 手动输入公众号信息
- 直接搜索后添加

### 查看订阅详情

当前订阅管理页是典型的“左列表 + 右详情”结构。选中一个订阅后，可看到：

- 公众号头像
- 名称和 ID
- 简介
- 状态
- 文章数量
- 最近文章时间范围

### 手动刷新文章

这是订阅管理最有价值的动作之一。它会直接调用更新接口，对指定公众号执行一次抓取。

注意：

- 有频率限制
- 依赖微信登录状态
- 不是简单刷新前端缓存

### 编辑和删除

支持修改公众号基础信息和状态，也支持删除订阅。

删除时要意识到一个事实：

- 订阅删除不等于所有历史文章一定立刻完全消失
- 还要考虑文章表和导出层是否有历史数据

## 4. 订阅到文章的真实链路

这条链路是理解项目的关键：

1. 用户添加订阅
2. 系统生成 `Feed` 记录
3. 记录 `faker_id` 等抓取关联信息
4. 定时或手动触发采集
5. 文章进入 `Article` 表
6. 后续才有标签、热点、RSS、消息任务

所以“订阅管理”并不是个孤立后台页，它决定了整个系统的上游质量。

## 5. 频繁遇到的几个误区

### 误区一：订阅只是给 RSS 用的

不对。RSS 只是订阅的一个输出形态，订阅本身是采集入口。

### 误区二：刷新订阅只是前端刷新

不对。当前的“刷新”会触发后端抓取逻辑，属于真实业务动作。

### 误区三：公众号名就是唯一标识

不对。当前系统里真正参与抓取和关联的还有：

- `id`
- `faker_id`

不能只看展示名。

## 6. 手动刷新时后台做了什么

在 [apis/mps.py](../../apis/mps.py) 里，手动刷新至少会做这些检查：

- 订阅是否存在
- 距离上次更新是否过近
- 微信 Session 是否有效
- 启动抓取线程
- 把新文章通过回调写入数据库

所以当用户说“点刷新报错”，不要先怀疑前端。优先查：

- 登录状态
- 限流提示
- 微信 token 是否过期

## 7. 和导入导出的关系

订阅管理还和导入导出打通了：

- 可导出公众号列表为 CSV
- 可导出为 OPML
- 可从 CSV 导入

这让它不仅是操作页，也是迁移和备份入口。
