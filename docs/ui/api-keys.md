# API 密钥管理

API Key 管理页用于给系统用户创建和维护可编程访问凭证。它面向的是“要让脚本、自动化流程或第三方程序访问 WeRSS API”的场景，不是给普通阅读用户用的。

## 1. 访问路径

- `/api-keys`

通常需要管理员权限或相应访问权限。

## 2. 这个页面能做什么

当前页面支持：

- 查看当前用户的 API Key 列表
- 创建新的 API Key
- 编辑名称、权限和启用状态
- 删除 API Key
- 重新生成 Key
- 查看使用日志

## 3. 页面结构

页面通常分为左右或上下两个主要区域：

### Key 列表

展示：

- 名称
- 权限
- 启用状态
- 最近使用时间
- 创建时间

### 详情与日志

选中某个 Key 后，可查看：

- 基础信息
- 使用日志
- 是否显示真实 Key 值

## 4. 权限字段怎么理解

当前前端里权限选项比较轻，主要是：

- `read`
- `read_write`

但后端实际保存的是字符串形式的权限字段，所以这里本质上是一个“权限串管理”页面，而不是完整 RBAC 控制台。

## 5. 创建时要注意什么

最重要的一点：

- API Key 的明文通常只在创建或重生成时完整返回一次

页面当前为了解决“看完就丢”的问题，会把新生成的 Key 通过本地存储暂时保留映射关系，用于当前浏览器后续显示。

这意味着：

- 换浏览器或清空本地存储后，旧明文不一定还能看见
- 但 Key 本身仍然有效，后端不会因为前端看不见而失效

## 6. 使用日志怎么看

API Key 日志主要用于回答这几个问题：

- 这个 Key 是否真的被调用过
- 调的是哪个接口
- 请求方法是什么
- 返回状态码是否异常

如果出现“某个自动化脚本说自己在调用，但系统没反应”，这页是第一批排查入口。

## 7. 常见操作建议

### 给内部脚本使用

建议单独创建一个名称明确的 Key，不要复用个人临时 Key。

### 给不同用途拆 Key

建议按用途拆开，例如：

- 只读同步
- 运维脚本
- 外部集成

这样日志和权限更清楚。

### 怀疑泄漏时

最稳的是直接重新生成或删除旧 Key，而不是继续复用。

## 8. 常见误区

### 误区一：创建完随时都能在页面重新看到完整 Key

不一定。页面只是做了本地缓存辅助展示，后端不会反复返回完整明文。

### 误区二：API Key 和登录 JWT 是一回事

不是。JWT 是登录态，API Key 是长期可编程凭证。

### 误区三：只有管理员才有意义

不完全对。管理员更常用，但普通用户如果系统允许，也可能需要为自己的自动化脚本创建 Key。
