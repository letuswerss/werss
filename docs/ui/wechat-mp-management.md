# 公众号管理

公众号管理页是一个更直接的管理视角。和“订阅管理”相比，它更像一个基础数据维护页，而不是日常采集操作页。

## 1. 访问路径

- `/wechat/mp`

通常需要 `wechat:manage` 相关权限。

## 2. 这页和订阅管理有什么区别

两者都围绕公众号，但侧重点不同：

- 订阅管理
  更偏日常使用、查看详情、刷新文章
- 公众号管理
  更偏基础记录维护和增删改

当前代码里，这个页面实际上仍然复用了订阅数据源，并没有另一套完全独立的 `mp` 管理后端。

## 3. 页面结构

### 列表区

当前页面展示的是一个表格，字段主要包括：

- 公众号 ID
- 名称
- 状态
- 最后同步时间
- 操作按钮

它不是“左列表右详情”的页，而是标准表格管理页。

### 操作区

每行支持：

- 编辑
- 删除

页面顶部支持：

- 添加公众号

## 4. 新增和编辑

弹窗表单主要字段包括：

- 公众号 ID
- 名称
- 封面图
- 简介
- 状态

新增时，本质上还是走创建订阅的逻辑；
编辑时，则走更新订阅的逻辑。

所以这里要理解一个现实：

- “公众号管理页”目前更多是管理 `Feed`/订阅记录的另一种 UI 入口
- 不是一个完全独立的公众号实体后台

## 5. 删除时要注意什么

当前页面删除调用的是删除订阅逻辑。

因此不能在文档里直接写成“删除公众号一定会删除所有相关文章”，因为这取决于后端删除实现，不能凭页面字面推断。

更稳妥的理解是：

- 它会删除该公众号对应的订阅记录
- 相关文章如何处理，要以实际后端删除逻辑为准

## 6. 适合什么场景

这页更适合：

- 管理员批量维护公众号基础信息
- 纠正名称、封面图、简介
- 快速禁用某个来源

不太适合：

- 做日常内容处理
- 观察某个订阅最近文章详情

这些更适合在订阅管理页做。

## 7. 常见问题

### 为什么这里和订阅管理的数据很像

因为当前实现本来就复用了订阅接口和数据。

### 为什么没有采集历史区

因为当前页面代码没有独立的采集历史面板。旧文档里那种“右侧详情 + 采集历史”写法并不符合现状。

### 为什么上传头像只看到 URL

页面当前支持上传封面图并把返回 URL 写回表单，本质上仍然是资源地址管理，不是复杂媒体库。

## 8. 建议怎么用

- 要看和刷文章：去 [订阅管理](./subscriptions.md)
- 要做基础信息维护：用这页

这样职责最清楚。
